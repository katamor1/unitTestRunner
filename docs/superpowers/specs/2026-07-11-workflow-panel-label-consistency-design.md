# Workflowパネル表現整合性デザイン

作成日: 2026-07-11  
対象: VS Code拡張の簡易表示・詳細表示

## 目的

Workflowパネルでは、工程見出しとボタンに同じ番号・同じ文言が重複し、完了済み工程でも初回実行時と同じボタン名が表示されている。簡易表示と詳細表示の両方で、状態・見出し・説明・操作ラベルの役割を統一し、現在の状況と次の操作を一目で判断できるようにする。

## 表現ポリシー

### 状態表示

簡易表示と詳細表示で、状態ラベルを次の3種類に統一する。

| 内部状態 | 表示 |
|---|---|
| `done` | `完了` |
| `current` | `次の操作` |
| `pending` | `未実施` |

`現在の推奨` は使用しない。

### 見出しとボタンの役割

- 見出しは工程・成果物を表す名詞句にする。
- ステップ番号は見出しだけに付ける。
- ボタンには番号を付けず、実際に行う操作を動詞で表す。
- 説明文は操作名を繰り返さず、確認対象と目的を短く示す。

例:

```text
完了
3. ビルド
生成・修正したテストをコンパイルし、リンク結果を確認します。

[ ビルドを再実行 ]
```

## 状態別アクションラベル

アクション種別ごとに、状態に応じて次のルールを適用する。

### コマンド実行

- `current` / `pending`: 初回実行ラベルを表示する。
- `done`: 再実行・再生成ラベルを表示する。
- Primary強調は `current` の主操作だけに適用する。
- `done` と `pending` の操作は通常ボタンとして表示する。
- 確認ダイアログが必要な危険操作の属性は維持する。

### レポート・ファイルを開く操作

完了状態でも「再確認」とはせず、常に「〜を開く」で統一する。ファイルを開く操作は状態を変更しないためである。

### 工程確定操作

`保存済みとして確定` は未完了工程だけに表示する。完了済み工程では状態ラベルの `完了` が同じ意味を担うため、確定ボタンは表示しない。

## 簡易表示

4ステップの文言を次に統一する。

| 見出し | 説明 | 初回・未完了ボタン | 完了後ボタン |
|---|---|---|---|
| `1. Quick Check` | 解析とテスト生成を行います。 | `Quick Checkを実行` | `Quick Checkを再実行` |
| `2. テストソース確認` | 入力値・期待値・スタブ設定を確認し、必要に応じて修正します。 | `テストソースを開く` | `テストソースを開く` |
| `3. ビルド` | 生成・修正したテストをコンパイルし、リンク結果を確認します。 | `ビルドを実行` | `ビルドを再実行` |
| `4. テスト実行` | 生成されたテストを実行し、結果レポートを確認します。 | `テストを実行` | `テストを再実行` |

結果・補助セクションのラベルは、状態に依存しないため現行の「〜を開く」「Full Gateへ進む」を維持する。

表示切替は `簡易` / `詳細` とし、「従来」は使用しない。

## 詳細表示

詳細表示でも同じ状態・ボタンポリシーを適用する。工程タイトルの番号は維持し、アクションボタンには番号を付けない。

主なコマンドアクションの完了後ラベルは次のとおり。

| 初回ラベル | 完了後ラベル |
|---|---|
| `現在関数を解析` | `現在関数の解析を再実行` |
| `選択関数を解析` | `選択関数の解析を再実行` |
| `テスト設計を生成` | `テスト設計を再生成` |
| `ハーネスを生成` | `ハーネスを再生成` |
| `dry-runを実行` | `dry-runを再実行` |
| `ビルドプローブを実行` | `ビルドプローブを再実行` |
| `テストを実行` | `テストを再実行` |
| `エビデンスを準備` | `エビデンスを再生成` |

`現在関数の解析を再実行` は、変更影響・既存テスト突合を行う別コマンド `変更後に再解析` と区別する。

レポートを開く操作は、完了後も以下のように変更しない。

- `dossierを開く`
- `確認リストを開く`
- `CSVを開く`
- `ビルドレポートを開く`
- `テスト実行レポートを開く`
- `エビデンスを開く`

## 実装境界

### WorkflowAction

アクション定義に完了後ラベルを明示できる任意プロパティを追加する。

```ts
repeatLabel?: string;
```

再実行可能なコマンドだけに `repeatLabel` を設定する。レポートを開く操作と補助操作には設定しない。

### レンダリング

アクション描画時に工程状態を渡し、表示ラベルとクラスを次の順で決定する。

1. `done` かつ `repeatLabel` がある場合は `repeatLabel` を使う。
2. それ以外は `label` を使う。
3. `primary` クラスは `current` の主操作だけに付ける。
4. `confirmStep` は `done` の場合は描画しない。
5. 実行中表示に使う `data-label` も画面上の動的ラベルと一致させる。

簡易表示と詳細表示は同じラベル解決関数を利用し、個別に文言分岐を持たない。

## テスト

最低限、以下を自動テストする。

1. 簡易表示でステップ番号が見出しだけにあり、ボタンに番号がない。
2. 完了済みQuick Check・ビルド・テストのボタンが「再実行」になる。
3. テストソースを開くボタンは完了後も同じラベルである。
4. 詳細表示の完了済みコマンドが `repeatLabel` を表示する。
5. 詳細表示の現在工程が `次の操作`、未実施工程が `未実施` と表示される。
6. 完了済み工程に `保存済みとして確定` が表示されない。
7. `data-label` とボタン表示が一致する。
8. パネルHTMLに「従来」「現在の推奨」が含まれない。

## 対象外

- Workflowの工程順序変更
- コマンド実行条件の変更
- pending工程の無効化
- build/test確認ダイアログの変更
- Full GateやQuick CheckのCLI処理変更
