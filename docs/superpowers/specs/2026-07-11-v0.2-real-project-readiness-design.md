# unitTestRunner v0.2 Real-Project Readiness 設計

作成日: 2026-07-11  
状態: 企画確定・実装未着手  
対象リポジトリ: `katamor1/unitTestRunner`  
対象リリース: `0.2.0`

## 1. 概要

`unitTestRunner` の次期版は、個別の解析機能をさらに増やすことよりも、既存の解析・ハーネス・ビルド・実行・回帰機能を実際のVC6/C90プロジェクトへ安全に導入できる状態へ引き上げることを優先する。

v0.2のテーマは **Real-Project Readiness** とする。

> VC6/C90プロジェクトを事前診断し、テストしやすい関数を選び、Quick Check、レビュー、変更影響確認、回帰実行までを一貫して進められる版にする。

v0.2では、次の4段階を正式なリリーススコープとする。

1. 品質基盤と公開契約の安定化
2. 実案件ワークスペースの事前診断
3. 依存ポリシーとテスト設計の構造化レビュー
4. Git差分を起点とする変更関数の一括Quick Checkと回帰スイート連携

実行カバレッジは実験機能として設計するが、v0.2のリリース条件には含めない。

本書はv0.2全体のアンブレラ設計である。各マイルストーンは独立した設計書、実装計画、PRへ分割し、単一の大型PRでは実装しない。

## 2. 背景

### 2.1 現在の到達点

現行実装は、当初の「関数単位dossier生成ツール」を超えて、以下まで到達している。

- DSW/DSP解析とsource membership判定
- 関数位置、シグネチャ、グローバル、副作用、呼び出し、分岐の解析
- 境界値、同値クラス、テストケース設計案の生成
- C90/VC6向けハーネス、スタブ、build workspaceの生成
- build probe、ビルドエラー解析、安全な補完ループ
- 生成テストの実行、結果解析、エビデンス生成
- dossier確定、レビュー成果物、traceability、readiness判定
- 再解析、既存テスト設計の照合、回帰選定
- Quick Check / Full Gate
- 複数関数の回帰スイート
- DSP/LIBからのリンク済み関数解決
- 呼び出し関係ごとの `real` / `stub` / `auto` 依存ポリシー
- 製品シンボルと衝突しないdispatcher/stub生成

このため、次の価値は「機能が存在すること」ではなく、「実案件でどの機能まで使えるかを短時間で判断し、迷わず運用できること」にある。

### 2.2 現在の課題

#### 品質ゲートの契約不一致

実行パーサーは、`UTR RUN` の後に `OK` / `FAILED` / `SKIPPED` がないケースを成功扱いせず、`inconclusive`、`crashed`、`timeout` として扱う。一方、古いテストには `UTR RUN` だけを成功と期待するものが残っている。現行契約へ統一し、mainの標準ゲートを常時GREENにする必要がある。

#### OS依存テストの境界が不明確

VS Code adapterにはWindowsパスを検証するテストが多い。Windows固有契約は `path.win32` 等で明示し、一般ロジックはホストOS非依存にする。正式なWindowsゲートに加え、Linux上の移植性ゲートを設ける。

#### 製品版数と成果物契約が実装に追いついていない

Pythonパッケージ、VS Code拡張、配布物は `0.1.0` のままだが、実装範囲は実行、回帰、依存ポリシーまで拡大している。また、JSON Schemaは公開dossierのトップレベル契約を中心としており、主要成果物全体の構造契約、生成版数、互換性方針は十分に形式化されていない。

#### 実案件での最初の入口が関数指定に偏っている

利用者は、対象関数を選ぶ前に、DSW/DSP、PCH、include、library、macro、function pointer、外部globalなどの難易度を把握する必要がある。現状は、関数を指定して解析してから問題が分かる。プロジェクト全体を先に診断し、適した候補関数を提示する入口が必要である。

#### 主要なレビュー判断がJSON直接編集に依存する

依存関係の `real` / `stub` / `auto`、ケース単位override、期待値、global初期値などは、生成JSONを直接編集する運用が中心である。機械生成部分と人間判断を分離し、stale更新を防止する構造化レビュー機構が必要である。

#### 変更関数の選択が手動である

再解析、回帰選定、suite実行の基盤は存在するが、Git差分から変更行を関数へ対応付け、複数関数を一括Quick Checkする公開ワークフローがない。

## 3. 目的

v0.2では次を実現する。

- mainの必須テストと契約テストを常時GREENにする
- 製品版数、artifact schema、generator versionの役割を明確にする
- VC6 workspace全体の自動化適合性を、関数解析前に評価できるようにする
- テストしやすい関数候補と、blocker / review itemの根拠を提示する
- ローカル環境、VC6 toolchain、outputRoot、encoding、DSW/DSP/LIBをまとめて診断する
- 依存ポリシー、外部global、テストケース、未解決事項をVS Codeで構造化レビューできるようにする
- 人間判断を生成物から分離し、再解析後も安全に保持する
- Git差分から変更関数と推移的影響関数を抽出する
- 変更関数を一括Quick Checkし、必要なものだけ回帰suiteへ登録する
- 本番ソースツリーを読み取り専用入力として扱う既存方針を維持する
- CLIを唯一の解析・検証ロジック層とし、VS Code adapterを薄い表示・操作層に保つ

## 4. 非目標

v0.2では次を正式スコープに含めない。

- 完全なCコンパイラフロントエンドへの置換
- 期待戻り値、期待global値、仕様判断の無審査自動確定
- 関数ポインタ、複雑なmacro、member callの強制的な自動書き換え
- 本番ソースへの生成コード、計測コード、設定変更の書き込み
- モジュール全体を1つのハーネスへ自動統合する機能
- リアルタイム性、32ms制約、疑似時間、疑似割込、hardware I/O再現
- 実機動作保証
- クラウドtelemetryやソース内容の外部送信
- バイナリ署名、インストーラー、Marketplace公開
- MC/DCを含む完全な実行カバレッジ

## 5. 方向性の比較と決定

### 案A: 実案件導入準備を優先する

内容:

- workspace assessment
- project-aware doctor
- 構造化レビュー
- Git差分からの一括Quick Check
- CI、schema、versionの安定化

利点:

- 既存機能の利用率と成功率を高める
- 実案件で失敗する理由を分類できる
- v0.3以降の優先順位を実データから決められる
- 本番非侵襲、dossier-first、CLI coreの方針を維持できる

欠点:

- 新しい解析アルゴリズムよりも基盤整備が中心で、見た目の派手さは小さい

### 案B: 実行カバレッジを優先する

内容:

- branch、switch、return、loop通過ログ
- test caseとcoverage itemの対応

利点:

- テストの実行到達度を明確に示せる
- 現行の設計カバレッジを実行証跡へ接続できる

欠点:

- build-readyへ到達できない実案件では価値が出ない
- instrumentationとVC6互換性の検証コストが高い

### 案C: モジュール単位へ拡張する

内容:

- function dossier集合化
- module dependency graph
- 共通global setup、共通stub、複数関数テスト

利点:

- 大規模案件の利用範囲を広げられる

欠点:

- 共有状態と依存関係が急増する
- 関数単位での成功・失敗要因を十分に収集する前に複雑度を上げることになる

### 決定

v0.2は案Aを採用する。

案Bは実験機能として独立したfeature flag配下に置き、v0.2のリリース条件から外す。案Cはworkspace assessmentと実案件pilotから得られたデータを基にv0.3以降で再評価する。

## 6. 目標ユーザーフロー

```text
VS CodeでVC6/C90プロジェクトを開く
  -> セットアップ診断を実行
  -> workspace assessmentを実行
  -> 推奨関数候補とblockerを確認
  -> 対象関数を選んでQuick Check
  -> 依存ポリシーとテスト設計を構造化レビュー
  -> Full Gate
  -> Git差分から変更関数を抽出
  -> 変更関数を一括Quick Check
  -> 選択した関数をsuiteへ登録
  -> 明示操作でbuild/testを実行
  -> evidenceを確定
```

CLI利用者も同じ処理をコマンドで再現できる。VS Code固有の状態を正本にしない。

## 7. リリーススコープ

### Milestone 0: 品質基盤と公開契約

Milestone 0は、他のv0.2機能に先行するリリースblockerである。

#### M0-1. 実行結果契約の統一

- `UTR RUN` のみをpassにしない
- 完了markerなし、非zero exit、timeoutを区別する
- 古い期待値を現行parser契約へ更新する
- designed caseとrunner outputのreconciliationを契約テストにする
- case isolationの回帰テストを必須化する

#### M0-2. OS依存テストの分離

- Windowsパス契約は `path.win32` または明示的なWindows fixtureで検証する
- host-independentなpath処理はWindows/Linuxの両方で同じ期待値にする
- Pythonの正式ゲートはWindowsを維持する
- VS Code adapterはWindows正式ゲートとLinux移植性ゲートを持つ
- Linuxでのみ意味を持つ差異をskipで隠さず、Windows固有契約として明示する

#### M0-3. versionとartifact contract

製品版数とartifact schema versionを分離する。

- 製品版数: `0.2.0`
- `generator_version`: 成果物を生成したunitTestRunnerの製品版数
- `schema_version`: artifact種別ごとの構造版数

既存artifactを無条件に一括で `schema_version: "0.2"` へ変更しない。構造互換のartifactは既存schema versionを維持し、非互換変更が必要なartifactだけ新schemaを導入する。readerは移行対象について旧版と新版の両方を受け入れる。

次をCIで同期検証する。

- `pyproject.toml`
- `src/unit_test_runner/__init__.py`
- `vscode/extension/package.json`
- VSIX名
- 配布文書内のversion表記

主要JSON artifactには、対応するschemaとcontract testを追加する。

対象:

- `function_dossier.json`
- `dependency_policy.json`
- `test_case_design.json`
- `build_workspace_report.json`
- `build_probe_report.json`
- `test_execution_report.json`
- `evidence_manifest.json`
- `suite_manifest.json`
- `suite_run_report.json`
- v0.2で追加するassessment / review / change artifact

#### M0-4. readinessの段階ゲート化

単一quality scoreを合否の根拠にせず、次の工程別gateを正本にする。

- `analysis_ready`
- `design_reviewed`
- `harness_ready`
- `build_ready`
- `execution_ready`
- `evidence_ready`

各gateは次を持つ。

```json
{
  "gate": "harness_ready",
  "status": "ready",
  "blockers": [],
  "review_items": [],
  "evidence": ["reports/function_signature.json"],
  "evaluated_at": "ISO-8601"
}
```

`status` は次のいずれかとする。

- `ready`
- `review_required`
- `blocked`
- `not_applicable`

既存の `quality_score` は後方互換の参考値として残せるが、workflow遷移や合否には使用しない。

#### M0-5. リリースworkflow

Windowsのtag workflowで次を実行する。

1. Python full test
2. CLI fixture smoke
3. VS Code compile/test
4. PyInstallerによるexe生成
5. exe fixture smoke
6. exeをVS Code拡張へ同梱
7. VSIX生成
8. VSIX内容検査
9. artifact SHA-256生成
10. GitHub Actions artifactとしてexe、VSIX、hashを保存

v0.2では自動公開や署名を行わない。

### Milestone 1: 実案件workspace assessment

#### M1-1. `assess-workspace`

新しい中心コマンドを追加する。

```powershell
unit-test-runner assess-workspace `
  --workspace D:\work\product `
  --dsw D:\work\product\Product.dsw `
  --out D:\work\unit_test_workspace\assessment
```

解析対象:

- DSWに含まれるprojectと直接依存
- DSP configuration、source membership、include、define、PCH、forced include
- LINK32 / LIB32、library path、依存project出力
- source encoding
- file/function数
- function signatureの解析可能性
- global read/write候補
- call種別
- function-like macro
- function pointer
- member call
- function address use
- unresolved include/macro/library
- dependency policyの自動解決可能性
- build workspace生成を妨げる要因

関数ごとに次を出す。

- project / configuration / source / function
- `automation_fit`
- blocker codeと根拠
- review item codeと根拠
-推奨Quick Check profile
- 参照artifactとsource location

`automation_fit` は次のいずれかとする。

- `good_candidate`
- `review_heavy`
- `blocked`
- `unknown`

不透明な100点満点ランキングを正本にしない。候補の並び順は、blocker数、review item数、構文の解析可能性、依存の一意性を使う決定的な規則とし、規則を文書化する。

出力:

```text
<out>/reports/workspace_assessment.json
<out>/reports/workspace_assessment.md
<out>/reports/function_candidates.csv
<out>/reports/assessment_diagnostics.json
```

代表的なJSON形:

```json
{
  "schema_version": "0.1",
  "generator_version": "0.2.0",
  "workspace": {
    "root": "D:/work/product",
    "dsw": "D:/work/product/Product.dsw"
  },
  "summary": {
    "projects": 3,
    "functions": 240,
    "good_candidates": 48,
    "review_heavy": 130,
    "blocked": 42,
    "unknown": 20
  },
  "functions": [
    {
      "source": "src/control.c",
      "function": "Control_Update",
      "project": "Control",
      "configuration": "Win32 Debug",
      "automation_fit": "good_candidate",
      "blockers": [],
      "review_items": [],
      "recommended_profile": "design"
    }
  ]
}
```

assessmentは問題を発見してもreport生成に成功した場合はexit 0とする。`status` を `assessed` または `assessed_with_blockers` として返す。入力不正、読取不能、出力不能だけを非zeroとする。

#### M1-2. project-aware `doctor`

既存 `doctor` にproject contextを追加する。

```powershell
unit-test-runner doctor `
  --workspace D:\work\product `
  --dsw D:\work\product\Product.dsw `
  --output-root D:\work\unit_test_workspace `
  --vcvars D:\VC98\Bin\VCVARS32.BAT
```

確認項目:

- Python version
- temp directory write
- sourceRoot read
- outputRoot write
- outputRootがsourceRoot外であること
- 長すぎるpathと予約文字
- CP932 read/write
- DSW/DSP参照の存在
- include directoryの存在
- unresolved DSP macro
- `cl.exe` / `link.exe` / `nmake.exe`
- `vcvars` 実行可否
- verification compiler
- library pathと主要libraryの存在

環境不足を検出しても診断reportを生成できた場合はexit 0とし、checkごとの `ok` / `warning` / `error` を返す。コマンド自体を実行できない入力不正だけを非zeroとする。

#### M1-3. workspace indexとcache

DSW/DSP、header include graph、source function index、library symbol indexを、外部workspaceのcacheへ保存する。

方針:

- 本番source treeへcacheを書かない
- cache rootは既定で `<outputRoot>/.unit-test-runner/cache/`
- file path、size、mtimeを高速な変更候補判定に使う
- 変更候補になった入力はSHA-256で再確認する
- final reportが参照した入力hashをartifactへ記録する
- cache hit/missと無効化理由をdiagnosticへ残す
- 読めないcacheは無視して再構築し、解析をinternal errorにしない
- cache schemaが非対応の場合は安全に再構築する

cacheは性能最適化であり、正しさの根拠にしない。

### Milestone 2: 構造化レビュー

#### M2-1. review model

既存 `prepare-review` を拡張し、Markdownに加えて機械可読なreview modelを生成する。

```text
reports/review_model.json
reports/review_model.md
```

review modelは次を統合する。

- dependency policy
- signature resolution
- external object binding
- test case design
- unresolved items
- readiness gates
- related evidence

review modelは生成専用であり、利用者は直接編集しない。

#### M2-2. 人間判断の分離

人間が確定した判断は、生成artifactとは別の `review_decisions.json` に保存する。

```json
{
  "schema_version": "0.1",
  "generator_version": "0.2.0",
  "base": {
    "workspace_id": "...",
    "dossier_sha256": "...",
    "review_model_sha256": "..."
  },
  "dependency_decisions": [
    {
      "callee": "ReadSensor",
      "configured_mode": "stub",
      "rationale": "hardware boundaryを分離するため",
      "review_status": "approved"
    }
  ],
  "test_case_decisions": [],
  "unresolved_item_decisions": []
}
```

保存対象:

- `real` / `stub` / `auto`
- case-level dependency override
- external object binding判断
- argument/global initial value
- stub return / side effect
- expected result
- review status
- rationale
- reviewer identifier。ローカル表示名でよく、外部送信しない
- decision timestamp

#### M2-3. `apply-review-decisions`

```powershell
unit-test-runner apply-review-decisions `
  --workspace D:\work\unit_test_workspace\Control_Update `
  --input D:\work\unit_test_workspace\Control_Update\reports\review_decisions.json
```

処理:

1. schema validation
2. workspace/function/source/configuration整合性確認
3. base hashによるstale check
4. decisionの意味検証
5. conflict preview生成
6. `review_decisions.json` 保存
7. dependency policyとtest designへの反映
8. 必要なharness/build artifactの再生成
9. readiness再評価
10. 変更artifact一覧をJSONで返す

staleなdecisionは自動適用せず、`review_decision_stale` として拒否する。`--force` は提供しない。利用者は最新review modelに対して再確認する。

#### M2-4. VS Code review editor

Workflow panelに次のsectionを追加する。

- 依存関数
- 外部global
- テストケース
- 未解決事項
- readiness gate

UIで表示する内容:

- 現在値
- 自動判定値
- 判定根拠
- source/evidence link
- 選択可能値
- rationale入力
- stale状態
- 適用後に再生成されるartifact

VS Code adapterはschema判断、policy解決、merge、regenerationを実装しない。CLIのreview modelを表示し、利用者入力を `apply-review-decisions` へ渡す。

適用前にはJSON差分と影響artifactのsummaryを表示する。build/test実行は従来どおり別操作と確認dialogを必要とする。

### Milestone 3: 変更関数の一括処理

#### M3-1. `scan-changes`

```powershell
unit-test-runner scan-changes `
  --workspace D:\work\product `
  --dsw D:\work\product\Product.dsw `
  --base origin/main `
  --out D:\work\unit_test_workspace\changes
```

`--head` を指定した場合はcommit/ref間を比較する。省略時はbaseと現在のworking treeを比較し、staged/unstaged変更と対象拡張子のuntracked fileを含める。

対象拡張子:

- `.c`
- `.h`
- `.dsw`
- `.dsp`
- `.lib`。実体変更を検出できるローカルpathだけ

処理:

- Git diffから変更fileとline rangeを抽出
- rename、add、deleteを識別
- `.c` の変更lineをfunction rangeへ対応付け
- `.h` 変更をinclude graphから関数へ逆引き
- define、DSP configuration、project membership変更を関連関数へ展開
- library変更をprovider関数利用箇所へ展開
- 判定不能な変更はfile/project単位へ保守的に拡大
- direct impactとtransitive impactを分離

出力:

```text
reports/change_set.json
reports/change_set.md
reports/impacted_functions.csv
```

各impactは次を持つ。

- function identity
- `direct` / `transitive` / `conservative`
- reason code
- confidence
- changed file / line
- related dependency/global/coverage item
- recommended action

#### M3-2. `quick-check-changes`

```powershell
unit-test-runner quick-check-changes `
  --change-set D:\work\unit_test_workspace\changes\reports\change_set.json `
  --profile design `
  --out D:\work\unit_test_workspace\change-runs
```

方針:

- 既定ではQuick Checkだけを実行する
- build/test executableは起動しない
- functionごとに独立workspaceを使う
- workspace index/cacheを再利用する
- function単位の成功/失敗を集約する
- 1件の失敗で他functionの解析を中断しない
- `--fail-fast` は提供できるが既定falseとする
- 結果から利用者が選択したfunctionだけをsuiteへ登録する

出力:

```text
reports/batch_quick_check.json
reports/batch_quick_check.md
reports/batch_function_results.csv
suites/changes-<change-set-id>/suite_manifest.json
```

#### M3-3. VS Code change workflow

VS Codeに次を追加する。

- `変更関数を検出`
- `変更関数をQuick Check`

## 10. 診断とエラー処理

### 10.1 診断コード

新機能は自然言語だけでなく、安定したdiagnostic codeを持つ。

カテゴリ:

- `environment_*`
- `workspace_*`
- `project_*`
- `source_*`
- `function_*`
- `dependency_*`
- `review_*`
- `change_*`
- `cache_*`
- `schema_*`

各diagnosticは次を持つ。

- code
- severity
- message
- evidence
- related path / function / project
- recommended action
- review_required

### 10.2 statusとexit code

- assessmentやdoctorは、問題を正しくreportできた場合はexit 0
- blockerの存在はJSON `status` とdiagnosticで表す
- 入力不正、読取不能、出力不能、schema破損など、コマンド目的を達成できない場合だけ非zero
- unsupported C constructは原則diagnosticへ変換し、内部例外にしない
- unexpected exceptionは既存のinternal error safety netで捕捉し、tracebackはverbose logへ出す
- batch commandはfunction単位の失敗を集約し、全体report生成に成功すれば結果を返す

### 10.3 stale protection

次のhashをdecision適用前に比較する。

- workspace identity
- source path
- function name
- configuration
- dossier SHA-256
- review model SHA-256
- dependency policy SHA-256
- test design SHA-256

不一致がある場合は自動適用しない。

## 11. 互換性方針

- 製品versionとartifact schema versionを分離する
- 既存CLI commandは削除しない
- `quick-check` aliasの現行挙動を維持する
- 既存 `function_dossier.json` の安定pathを維持する
- finalizationは公開dossierを互換的に拡張し、別shapeへ置換しない
- 旧 `review_decisions` が存在しないworkspaceは通常どおり動作する
- 新しいdecision artifactはopt-inから開始し、VS Code editor利用時に生成する
- readerは移行対象artifactの旧schemaを受け入れる
- writerはartifactごとに定義した最新schemaだけを生成する
- schema migrationが必要な場合は明示commandまたは読み込み時のin-memory変換とし、元fileを黙って上書きしない

## 12. 安全性とprivacy

- 本番source、header、DSW、DSP、LIBを変更しない
- outputRootがsourceRoot内の場合はwarningと明示確認を要求し、assessment/doctorでは安全な外部pathを推奨する
- review decision適用は生成workspace内だけを変更する
- build/test executableの実行は明示optionまたはVS Code confirmationを必要とする
- Git commandは対象workspace内だけで実行する
- credential、email、source内容、利用統計を外部送信しない
- reviewer identifierはローカル表示用文字列とし、外部directory照会を行わない
- cacheとreportにsource全文を重複保存しない。必要なsource location、hash、要約を優先する
- logへ環境変数値を無条件に出さない

## 18. v0.3判断基準

v0.3では、workspace assessmentと実案件pilotの結果に基づき、次のいずれかを主テーマにする。

### 実行カバレッジを優先する条件

- build-readyへ到達するfunctionが十分に存在する
- 主な課題が「実行したがcoverage証跡が不足」である
- instrumentationによるVC6互換リスクを限定できる

### モジュール単位を優先する条件

- function単位では成功するが、共通global/setup/stubの重複が主要コストである
- 複数functionをまたぐ仕様観点が多数ある
- module内dependency graphを一意に構築できる案件が多い

### 解析精度改善を優先する条件

- blockerの多数がmacro、function pointer、conditional compilation、typedef解析に集中する
- build以前にanalysis-readyへ到達できないfunctionが多い

この判断により、次期テーマを印象ではなく実測されたblocker分布から選択する。

## 19. 最終決定

unitTestRunner v0.2は、**Real-Project Readiness** を正式テーマとする。

最初に品質ゲート、version、schema、readiness契約を安定させる。その後、`assess-workspace` を看板機能として実装し、プロジェクト全体からテストしやすい関数とblockerを提示する。続いて、人間判断を `review_decisions.json` に分離した構造化review workflowを追加し、最後にGit差分から変更関数を一括Quick Checkしてsuiteへ接続する。

実行カバレッジとモジュール単位拡張は急がない。v0.2で実案件の失敗理由、review量、build到達率を計測し、そのデータをv0.3の優先順位へ使用する。
